# Threat model and at-rest key protection

The app defends against (A) a lost/stolen **locked** device, (B) offline forensic disk extraction, and (D) cloud/adb backup exfiltration. It explicitly does **not** defend against (C) malware or a rooted device running as-or-above the app — once code executes with the app's identity it can read the Keystore-wrapped blob and hook process memory; this is the industry-standard line (Bitwarden, 1Password, KeePassDX make the same concession).

Consequence: the GPG and SSH private keys are stored via `AesGcmCryptoStore` with `setUserAuthenticationRequired(false)` so that `BootReceiver`, the autofill service, and background git sync can read them without user presence. This is deliberate, not an oversight. Under threat B the at-rest security of the GPG key therefore reduces to the strength of the GPG passphrase (its OpenPGP S2K protection); the per-use biometric gate lives on the **session/passphrase cache** (`AndroidBiometricCryptoStore`), not on the key blob. Backup exfiltration (D) is closed by `allowBackup=false` — which is the permanent answer, not a placeholder: GitHub is the source of truth for the store, and all local state (GPG key, SSH key, cloned repo) is re-importable or re-clonable, so there is nothing worth backing up and a backup would only leak the wrapped keys. The stub `backup_rules.xml` / `data_extraction_rules.xml` are intentionally inert.
