AppPreferencesTest — Implementation Notes

Task: Write unit tests for AppPreferences per spec.
Date: 2026-05-22

Design Decisions

DECISION Refactor AppPreferences to accept DataStore<Preferences> directly

The spec mandates JUnit5 unit tests using PreferenceDataStoreFactory.create(). The original AppPreferences constructed its DataStore internally via a Context-extension property (preferencesDataStore delegate). JUnit5 unit tests run on the JVM without an Android runtime, so there is no Android Context available. To satisfy the spec without Robolectric, the constructor was changed to accept DataStore<Preferences> directly, making the class context-free and trivially testable.

DECISION Added PreferencesModule Hilt module

Extracting Context from AppPreferences moves the responsibility for constructing the DataStore to DI. A new PreferencesModule provides a singleton DataStore<Preferences> using the original preferencesDataStore("app_prefs") delegate. All existing callers (ViewModels, MainActivity, etc.) continue to receive the same injected AppPreferences singleton — no callsite changes required.

DECISION One shared TestScope(UnconfinedTestDispatcher()) per test method, recreated in @BeforeEach

The DataStore's coroutine scope must outlive the runTest block (it is used as the DataStore's background scope). Creating a fresh TestScope per test in @BeforeEach and cancelling it in @AfterEach gives each test an isolated scope and avoids scope leaks. UnconfinedTestDispatcher causes DataStore I/O to run eagerly without explicit advanceUntilIdle() calls.

Deviations from Spec

DEVIATION AppPreferences.kt modified (not mentioned in spec)

The spec lists only one file to create. However, the original implementation cannot be exercised by JUnit5+PreferenceDataStoreFactory without a Context. The minimal source change (swap constructor param) was necessary to satisfy all other spec constraints simultaneously. Alternatives are discussed in Tradeoffs below.

DEVIATION PreferencesModule.kt created (not mentioned in spec)

Required to restore prod DI after removing Context from AppPreferences. Without it, Hilt would fail to satisfy the DataStore<Preferences> dependency at compile time.

Tradeoffs

Approach Pros Cons Chosen?
Inject DataStore<Preferences> (implemented) Pure JVM tests; no Robolectric; idiomatic DI; AppPreferences has no Android dep Requires touching prod source + adding a Hilt module ✅
Robolectric + JUnit4 (existing pattern in codebase) No prod refactor needed; matches existing test style Violates spec (JUnit5 mandated); DataStore singleton per context causes cross-test bleed unless manually cleared ✗
Robolectric + JUnit5 (@ExtendWith(RobolectricExtension)) JUnit5 compliant; no prod refactor Requires robolectric-junit5 dep not in project; still has DataStore singleton bleed risk; more complex setup ✗
Internal/package-private secondary constructor for tests Avoids DI changes Test-only code in prod source; non-idiomatic Kotlin DI pattern ✗

Open Questions

  • OPEN Test 7 covers all 6 spec prefs but not sessionLastTouched. The source has 7 prefs; the spec enumerates only 6 (no sessionLastTouched in defaults list or round-trip requirement). Tests follow the spec exactly. Confirm whether sessionLastTouched should be included in tests 1, 7, and/or 9.
  • OPEN Should testScope.cancel() flush pending DataStore writes before deletion? With UnconfinedTestDispatcher, writes are synchronous so this is not an issue in practice. If the dispatcher changes, teardown may need an explicit advanceUntilIdle() before cancel.
  • OPEN Turbine not used. The spec tests only the latest emitted value (first()). Turbine would add value for testing emission sequences (e.g., verifying intermediate states during clearAll). Include if emission ordering matters.
