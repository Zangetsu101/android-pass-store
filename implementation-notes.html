GpgImportViewModelTest — Implementation Notes

File: app/src/test/java/com/zangetsu101/pass/onboarding/GpgImportViewModelTest.kt

Design Decisions

Decision — every vs coEvery for importGpgKey

GpgKeyOperations.importGpgKey is a regular (non-suspend) function annotated with @Throws(KeyImportError::class). It is called inside withContext(Dispatchers.IO) in the ViewModel, but that doesn't make the function itself a suspend function. MockK's every is correct here; coEvery would also work but is semantically wrong.

Decision — relaxed = true on AppPreferences mock

AppPreferences is a concrete class (not an interface) with a suspend function setGpgImported. Using relaxed = true means MockK auto-stubs all methods, preventing UnstubbedMockKException on any calls not explicitly set up. In the success test, coJustRun is still explicitly set for clarity and to enable coVerify.

Decision — Shared StandardTestDispatcher between setMain and runTest

The same testDispatcher instance is passed to both Dispatchers.setMain() and runTest(testDispatcher). This ensures viewModelScope (which uses Dispatchers.Main) and the test coroutine share the same TestCoroutineScheduler, so advanceUntilIdle() drains all pending coroutines on both sides.

Decision — Direct constructor instantiation of GpgImportViewModel

Although the ViewModel is annotated with @HiltViewModel, Hilt injection is only needed at runtime. In unit tests the constructor is public and accepts the two dependencies directly, so GpgImportViewModel(cryptoOperations, appPreferences) works without any Hilt test infrastructure.

Deviations from Spec

Deviation — ViewModel modified to expose internal var ioDispatcher

The spec asks only for test code, but the ViewModel hard-codes Dispatchers.IO. With StandardTestDispatcher, withContext(Dispatchers.IO) dispatches to the real thread pool; advanceUntilIdle() drains only the test scheduler and returns before the IO continuation posts back — all three coroutine tests failed with stale state. Fix: added internal var ioDispatcher: CoroutineDispatcher = Dispatchers.IO to the ViewModel and set it to testDispatcher in @BeforeEach. Production behaviour is unchanged (default stays Dispatchers.IO; Hilt does not inject it). Adding it as a constructor parameter was rejected because Hilt treats every constructor parameter as a required binding and fails with MissingBinding for CoroutineDispatcher.

Deviation — coJustRun instead of coEvery { … } just Runs

The spec does not prescribe a specific MockK idiom. coJustRun { appPreferences.setGpgImported(true) } is the idiomatic shorthand for stubbing a suspend function that returns Unit; it is equivalent to coEvery { … } just Runs but more concise.

Deviation — assertNotNull for error message, not exact string match

Spec says "gpgImportError set" without specifying the exact string. Tests verify assertNotNull(state.gpgImportError). Exact string matching would couple tests to KeyImportError message literals, making them brittle to wording changes.

Tradeoffs

Tradeoff — StandardTestDispatcher vs UnconfinedTestDispatcher

Spec mandates StandardTestDispatcher. This dispatcher does not execute coroutines eagerly — manual advanceUntilIdle() is required. The risk is a race between withContext(Dispatchers.IO) (real thread pool) and advanceUntilIdle() returning too early. In practice this is safe because MockK stubs return synchronously, so the IO thread completes and posts the continuation back to Main before the scheduler checks for idle. UnconfinedTestDispatcher would eliminate this concern entirely but violates the spec.

Tradeoff — Mocking concrete AppPreferences vs extracting interface

AppPreferences has no interface. Options: (a) mock the concrete class with MockK (chosen — zero code changes), (b) introduce an AppPreferencesInterface and update the ViewModel. Option (a) keeps the change minimal and self-contained. Option (b) is better long-term but out of scope for this task.

Open Questions

Question — IO dispatcher substitution (resolved)

Originally flagged as a potential flakiness risk. Confirmed as the root cause: tests 2–4 all failed because advanceUntilIdle() returned before the real IO dispatcher posted the continuation back to Main. Resolved by adding internal var ioDispatcher to the ViewModel (see Deviations).

Question — setGpgKeyFromBytes coverage

The spec does not include tests for setGpgKeyFromBytes, which also launches a coroutine and uses Dispatchers.IO. The armoring path (non-PGP-header bytes) and error path (KeyImportError thrown by armorGpgKey) are untested. Confirm whether these are intentionally deferred.
