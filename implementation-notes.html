Design Decisions

DECISION Shared testDispatcher instance across setMain and runTest

StandardTestDispatcher() is created once as a class field and reused in both Dispatchers.setMain(testDispatcher) and runTest(testDispatcher). This ensures both the test body and viewModelScope.launch coroutines share the same TestCoroutineScheduler, so advanceUntilIdle() inside runTest actually drains the ViewModel's pending coroutines. Using two separate dispatcher instances would give two separate schedulers and advanceUntilIdle() would not advance the ViewModel's work.

DECISION ViewModel created in @BeforeEach, not inside runTest

The ViewModel's init { loadStatus() } fires at construction. Creating it in @BeforeEach (outside runTest) queues the init coroutine on testDispatcher but does not run it yet — the scheduler isn't being advanced. Each test then calls advanceUntilIdle() first to explicitly settle the init, giving clean per-test control over when the init coroutine completes.

DECISION passStore mocked with relaxed = true

PassStore.buildIndex() is not a suspend function and returns List<PassEntry>. With relaxed = true, MockK returns an empty list without requiring explicit setup. This keeps test 3 (pull success) clean — only the interactions worth asserting need explicit setup.

Deviations from Spec

DEVIATION Test 2: loadStatus() called on existing VM, not a fresh one

The spec says "loadStatus() when syncStatus() throws". The most natural reading is to test the method in isolation. Rather than constructing a second ViewModel where the init itself throws (which conflates init behavior with loadStatus behavior), the approach is:

  1. Settle the init with advanceUntilIdle() (good mock, init succeeds).
  2. Reconfigure the mock to throw.
  3. Call viewModel.loadStatus() explicitly.

This tests exactly what the spec describes — what loadStatus() does when syncStatus() throws — without interference from init behavior.

Tradeoffs

TRADEOFF StandardTestDispatcher vs UnconfinedTestDispatcher

StandardTestDispatcher UnconfinedTestDispatcher
Coroutine execution Manual — must call advanceUntilIdle() Eager — runs immediately on launch
Control High — can inspect state between steps Low — state changes happen inline
Spec requirement ✓ Explicitly required ✗

StandardTestDispatcher chosen per spec. Also better practice for ViewModel tests because it lets you inspect state before a coroutine runs (e.g., pulling=true intermediate state), though the current tests don't assert on intermediate states.

TRADEOFF JUnit5 (org.junit.jupiter) vs JUnit4 (org.junit)

Existing tests in the project (GitSyncImplTest, PassStoreImplTest) use JUnit4 annotations (@Before, @Test from org.junit). The spec explicitly requires JUnit5. The build is configured with useJUnitPlatform() and both junit.jupiter.api + junit.vintage.engine are in deps, so JUnit4 and JUnit5 tests coexist. Used JUnit5 (@BeforeEach, @AfterEach, org.junit.jupiter.api.Test, org.junit.jupiter.api.Assertions.*) per spec.

Open Questions

OPEN Should Test 4 verify pullError message content?

The spec says "pullError set" — currently asserted with assertNotNull(state.pullError). The ViewModel code is e.message ?: "Pull failed" so the error would be "Connection reset". The test doesn't assert the exact string. Confirm whether exact message content matters for the test.

OPEN Intermediate pulling=true state — assert or skip?

When pull() is called, the ViewModel sets pulling=true synchronously before launching the coroutine. This is observable state between pull() and advanceUntilIdle(). The spec doesn't list this as a test case. Confirm whether a Test 5 for intermediate state is desired.

OPEN pullSuccess reset on next loadStatus() — tested?

After a successful pull, if the user triggers a refresh, pullSuccess is not reset by loadStatus() (only by the next pull() call). This may be intentional UI behavior. Worth confirming and possibly adding a regression test.
